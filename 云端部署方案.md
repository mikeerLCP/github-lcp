## 一、项目理解（确认我明白了你要我干什么）

### 当前状态

| 部件 | 形态 | 位置 |
|------|------|------|
| 管理端（6个模块） | Tkinter 桌面 GUI | `商管原型\main.py` + 6个 module_*.py |
| 商户门户 | Flask Web（端口5050） | `商管原型\merchant_portal.py` |
| 数据层 | MySQL + JSON 双后端 | `商管原型\db.py` + `utils.py` |
| 用户系统 | Tkinter 登录窗口，MD5 密码，角色权限 | `login_window.py` + `module_user_management.py` |

### 目标

1. **全部功能 Web 化** — 6 个管理模块 + 商户门户全部走浏览器
2. **部署到云端** — 服务器上跑，任何设备都能用
3. **多租户** — 集团管理员、二级公司管理员、商户 三种角色在同一套系统里各看各的数据
4. **UI 风格** — 参考商户门户原型的卡片式设计风格

## 三、云端架构设计

### 3.1 技术选型

| 层次 | 选型 | 理由 |
|------|------|------|
| **Web 框架** | **Flask** | 已有 merchant_portal.py 的成功经验；轻量，适合中型系统；Jinja2 模板与现有风格无缝衔接 |
| **前端渲染** | **服务端渲染 (SSR) Jinja2** | merchant_portal.py 已验证可行；零前端框架学习成本；SEO 和性能对管理后台来说足够 |
| **前端增强** | **原生 JS + htmx（可选）** | htmx 可实现表格内编辑、分页加载等 AJAX 交互，不引入 React/Vue 重量级框架 |
| **数据库** | **MySQL 8.0** | 已有完整 schema、数据、驱动（pymysql），直接部署 |
| **认证** | **Flask-Login + Session** | 简单可靠，merchant_portal.py 已有 session 模式 |
| **部署** | **Gunicorn + Nginx** | 标准的 Python Web 部署栈 |
| **图表** | **Chart.js** | 轻量 JS 图表库，替代 matplotlib 在浏览器中渲染 |

## 九、关键决策说明

### 为什么选 Flask 而不是 FastAPI/Django？

- **Flask**：merchant_portal.py 已有成功案例；Jinja2 模板与 SSR 方式完美匹配；轻量，对于 7 个页面模块的管理系统够用
- **FastAPI**：更适合 API 优先 / SPA 前端，但我们走 SSR 路线
- **Django**：太重，ORM 学习成本高，现有 pymysql 代码需要全部重写

### 为什么走 SSR 而不是 SPA（React/Vue）？

- 现有 merchant_portal.py 是 SSR，已验证可行
- 管理后台不需要 SPA 的客户端路由和复杂状态管理
- 团队维护成本极低：HTML + Jinja2 + 一点原生 JS
- 适量 AJAX（htmx）可以搞定表格内编辑、筛选等交互

### 为什么用行级过滤而不是多数据库？

- 同一集团旗下商场共享商铺字典（业态分类等）
- 集团管理员需要跨商场查看（如果分库，JOIN 就废了）
- pymysql 代码零修改

## 十一、总结

**一句话：** 把 Tkinter 的 UI 层换成 HTML+CSS+JS，业务逻辑和数据库层原封不动搬过来，用一个 Flask 应用同时服务管理端（`/admin`）和商户端（`/portal`），通过角色权限实现数据隔离，部署到云服务器。

**核心原则：**
